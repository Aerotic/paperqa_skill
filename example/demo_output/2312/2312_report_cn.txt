1 核心问题：大型语言模型（LLM）固有的非确定性导致其在移动端任务自动化中缺乏可靠的任务记忆，难以高效重复执行已学任务。

2 方法创新：提出 MobileGPT 系统，通过类人应用记忆架构（Human-like App Memory）和三阶段推理结构（Explore-Select-Derive）增强LLM的任务复现能力，并引入双策略故障处理机制（Dual-Strategy Failure Handling）。

3 关键效果：在80个任务（160条指令）的自定义数据集上，MobileGPT在热启动（Warm-start）场景下达到98.75% 的步骤准确率，任务成功率82.5%；相较于基线方法，延迟降低62.5%，成本降低68.8%；屏幕分类在269个应用屏幕中仅出现3个假阳性，0个假阴性。

问题与挑战

移动端任务自动化面临的核心挑战是LLM的非确定性（Non-determinism）——即同一任务在不同执行轮次中可能产生不同输出，导致系统无法可靠地复现已学习过的操作流程。现有方法（如AutoDroid、AppAgent）缺乏有效的任务记忆机制，每次执行都需从头推理，造成计算资源浪费和用户体验下降。此外，移动应用界面（UI）的动态变化和任务参数的多样性（如不同收件人、不同搜索关键词）进一步增加了系统稳定复现任务的难度。

方法概览

MobileGPT提出了一种冷启动-热启动（Cold-start/Warm-start）双阶段工作流。冷启动阶段通过探索（Explore）、选择（Select）和推导（Derive）三阶段学习新任务，构建层次化记忆；热启动阶段则利用已存储的记忆，通过子任务槽填充（Sub-task Slot-filling）和上下文适配（In-context Adaptation）快速复现任务。系统采用GPT-4-turbo进行屏幕理解和推理，GPT-3.5-turbo执行槽填充，并将计算负载卸载至服务器，移动端平均功耗仅330.38 mW。

架构流程

1. 离线探索阶段：通过随机探索器（Random Explorer）和用户轨迹监控器（User Trace Monitor）自动发现应用页面，平均每应用发现50个页面（覆盖所需页面的89.65%），成本$10.78。
2. 冷启动学习阶段：用户演示新任务 → Explore阶段（LLM按屏幕收集子任务） → Select阶段（LLM选择最相关的子任务，包括全局子任务如“读取屏幕”和“完成”） → Derive阶段（LLM逐步选择低层动作如点击/输入/滚动） → 将任务结构存入三级层次记忆（任务-子任务-动作）。
3. 热启动执行阶段：用户输入自然语言指令 → 应用启动推荐（基于Google Play描述的向量数据库推荐Top-3应用） → 子任务槽填充（用GPT-3.5-turbo填充参数） → 上下文适配（从记忆检索匹配的任务结构） → 逐步执行动作。
4. 故障处理阶段：执行中若出错 → 自纠正（基于规则的反馈如“不存在索引i的UI元素”追加至提示词） → 若自纠正失败 → 人在回路（Human-in-the-loop）任务修复（系统可用性评分SUS 64.6）。

关键效果

- 热启动步骤准确率：98.75%，接近完美
- 热启动任务成功率：82.5%，整体成功率82.16%
- 延迟与成本：热启动延迟降低62.5%，成本降低68.8%（相较于基线）
- 记忆命中率：因应用而异（Telegram 98.6%，Gmail 70.6%）
- 屏幕分类：269个屏幕中仅3个假阳性，0个假阴性

三阶段推理结构（Explore-Select-Derive）

核心思想：将移动端任务完成过程分解为三个逻辑上独立的推理阶段，而非让LLM一次性生成所有动作。Explore阶段负责按屏幕收集所有可能的子任务；Select阶段让LLM从候选子任务中选择最相关的一个（包括全局子任务如“读取屏幕”和“完成”）；Derive阶段则逐步推导出具体的低层动作（点击、输入、滚动）。这种分解降低了单次推理的复杂度，使LLM能更专注于每个阶段的特定目标。

与已有工作的区别：传统方法（如AutoDroid）采用“直接推导”（Derive-only）结构，要求LLM一步到位地从屏幕状态映射到动作序列。这种端到端方式在面对复杂多步任务时容易出错，且无法有效利用任务的结构化信息。MobileGPT的三阶段结构将任务分解为可管理的子问题，每个阶段都有明确的输入输出规范。

技术实现：使用GPT-4-turbo作为核心推理引擎，分别处理三个阶段。Explore阶段输入当前屏幕的UI元素描述，输出该屏幕支持的所有子任务列表；Select阶段输入子任务列表和任务目标，输出选中的子任务；Derive阶段输入子任务和UI状态，输出具体动作。消融实验显示各阶段准确率分别为：Explore 96.4%、Select 96.2%、Derive 99.1%，证明该分解策略的有效性。

有效性证据：消融研究对比了三阶段结构与Derive-only基线，三阶段结构在热启动步骤准确率上达到98.75%，显著优于基线。各阶段的高准确率表明，将复杂推理任务分解为多个简单子任务能有效提升LLM的决策可靠性。

类人应用记忆与三级层次记忆架构

核心思想：模仿人类记忆的组织方式，将应用操作知识按任务（Task）、子任务（Sub-task）、动作（Action）三级层次化存储。任务层记录完整任务的目标和参数；子任务层按应用屏幕分组，记录该屏幕下可执行的子任务集合；动作层存储具体的UI交互指令。这种结构支持跨任务共享（不同任务可能共享相同的子任务序列），实现高效的存储和灵活的回忆。

与已有工作的区别：现有系统（如AppAgent）通常将任务记忆存储为扁平化的动作序列，缺乏结构化组织。当任务数量增多时，检索效率急剧下降，且无法利用任务间的共性。MobileGPT的层次记忆允许系统通过子任务匹配快速定位相关记忆，并通过参数槽填充（Slot-filling）处理任务变体。

技术实现：记忆以JSON格式存储，每个任务包含任务ID、目标描述、参数槽定义和子任务序列。子任务节点按应用屏幕分组，同一屏幕下的子任务共享UI上下文。热启动时，系统通过子任务槽填充（使用GPT-3.5-turbo）将用户指令中的参数映射到记忆中的槽位，然后检索匹配的任务结构。记忆命中率因应用复杂度而异：Telegram达98.6%，而Gmail仅70.6%，表明邮件类应用的参数多样性对记忆检索构成更大挑战。

有效性证据：热启动场景下98.75%的步骤准确率和82.5%的任务成功率直接证明了层次记忆的有效性。延迟降低62.5%和成本降低68.8%则量化了记忆复用带来的效率提升。跨任务共享能力在实验中表现为：当多个任务共享相同子任务序列时，系统无需重新学习即可复用。

双策略故障处理机制

核心思想：针对LLM在任务执行中可能出现的各种错误，设计两种互补的纠正策略。自纠正（Self-correcting）通过规则生成的反馈信息（如“不存在索引i的UI元素”、“你已经循环了相同屏幕X次”）追加到LLM提示词中，让模型自行调整行为。人在回路（Human-in-the-loop, HITL）则作为兜底机制，当自纠正失败时，允许用户手动干预修复Explore/Select/Derive阶段的错误。

与已有工作的区别：大多数移动端自动化系统（如AutoDroid）缺乏显式的错误处理机制，一旦LLM产生错误动作，任务即告失败。MobileGPT将错误处理作为系统的一等公民设计，通过反馈循环让系统具备自我修正能力，同时保留人工干预的接口以处理复杂或意外情况。

技术实现：自纠正模块在每次动作执行后检查执行结果，若检测到错误（如UI元素不存在、陷入循环），则生成结构化反馈信息并重新调用LLM。HITL模块提供一个图形化界面，允许用户查看当前任务状态、修改子任务选择或直接编辑动作序列。系统可用性评分（SUS）为64.6，表明HITL接口的易用性处于中等偏上水平。

有效性证据：虽然论文未提供自纠正成功率的单独数据，但整体任务成功率82.16%（热启动82.5%）表明故障处理机制有效挽救了部分失败任务。HITL的SUS评分64.6表明该机制在实用性和用户接受度上达到了可接受水平。

离线探索与应用页面发现

核心思想：在用户使用前，通过自动化探索手段预先发现应用的所有可能页面和UI状态，构建完整的应用知识库。采用随机探索器（Random Explorer）随机点击UI元素，结合用户轨迹监控器（User Trace Monitor）记录真实用户的操作路径，两者互补以最大化页面覆盖率。

与已有工作的区别：现有系统通常依赖用户演示或在线探索来学习应用结构，这导致冷启动阶段需要大量用户交互。MobileGPT的离线探索将页面发现过程自动化，用户只需在冷启动阶段演示任务流程，无需手动探索应用的所有功能。

技术实现：随机探索器在应用内执行随机点击、滑动等操作，记录每次操作后的屏幕状态。用户轨迹监控器则从用户日常使用中收集操作序列。两者结合后，系统对每个应用平均发现50个页面，覆盖任务所需页面的89.65%。整个探索过程的货币成本仅为$10.78（主要来自LLM API调用）。

有效性证据：89.65%的页面覆盖率表明离线探索能有效覆盖绝大多数任务所需页面。$10.78的低成本证明该方法在经济上可行。屏幕分类模块在269个屏幕中仅出现3个假阳性、0个假阴性的结果，进一步验证了探索阶段收集的页面数据质量高、分类准确。
